# 设计文章库本地管理工具

这个工具用于在浏览器中维护个人网站的设计文章库。它只在本机运行，负责修改仓库中的 `data/reading-library.json`；公开网站只展示其中状态为“已发布”的文章。

## 运行条件

- 已安装 Node.js。
- 已在 VS Code、Cursor 或终端中打开网站仓库根目录。

可以运行下面的命令确认 Node.js 是否可用：

```bash
node --version
```

## 启动工具

在仓库根目录运行：

```bash
npm run editor
```

工具会自动打开浏览器。如果没有自动打开，可以手动访问：

- 管理工具：<http://127.0.0.1:4173/>
- 公开页面本地预览：<http://127.0.0.1:4173/reading/>

如需启动服务但不自动打开浏览器：

```bash
npm run editor -- --no-open
```

完成编辑后，在终端中按 `Ctrl+C` 停止服务。

## 添加并发布一篇文章

1. 点击“新建文章”。
2. 手动填写原文链接、标题和推荐理由。
3. 选择一个主分类。
4. 根据需要选择多个 Tag。
5. 点击“保存”，文章会先保存为草稿。
6. 检查内容无误后点击“发布”。
7. 打开公开页面本地预览，确认文章展示正常。
8. 在 VS Code 或 Cursor 中检查 Git 变更，提交并推送到 GitHub。

管理工具中的“发布”只会把文章状态改为 `published`，不会自动执行 Git 提交或推送。只有推送仓库并完成 GitHub Pages 部署后，线上网站才会更新。

## 编辑已有文章

在左侧文章列表中选择一篇文章，然后修改内容并点击“保存”。

文章列表支持：

- 按标题搜索；
- 按草稿或已发布状态筛选；
- 按主分类筛选；
- 按 Tag 筛选。

如果表单存在未保存修改，切换文章或关闭页面时会出现提醒。

## 草稿、发布和撤回

- 新文章默认是草稿，公开页面不会展示。
- 点击“发布”后，文章会进入公开数据列表。
- 已发布文章可以点击“撤回为草稿”，从公开页面中移除。
- 草稿和撤回操作都需要提交并推送 Git 变更后，线上网站才会同步变化。

草稿与已发布文章保存在同一个 JSON 文件中。公开页面会过滤草稿，但直接查看公开仓库或 JSON 文件的人仍可能看到草稿内容，因此不要在草稿中保存敏感信息。

## 管理主分类

点击页面右上角的“管理分类”。

可以执行：

- 创建新分类；
- 修改分类名称；
- 查看分类关联的文章数量；
- 删除未使用的分类；
- 将使用中的分类迁移到另一个分类后再删除。

每篇文章必须有一个主分类，因此不能直接删除仍被文章使用的分类，也必须至少保留一个分类。

## 管理 Tag

点击页面右上角的“管理 Tag”。

可以执行：

- 创建新 Tag；
- 修改 Tag 名称；
- 查看 Tag 关联的文章数量；
- 将一个 Tag 合并到另一个 Tag；
- 删除 Tag。

删除正在使用的 Tag 时，工具会显示受影响的文章数量，并要求输入 Tag 名称进行确认。删除后，该 Tag 会同时从所有相关文章中移除。

## 数据文件

所有内容保存在：

```text
data/reading-library.json
```

一般情况下应通过管理工具修改它，不建议手动编辑。如果确实需要手动修改，完成后可以运行测试检查数据结构：

```bash
npm test
```

服务端会在保存前校验数据，并采用临时文件加原子替换的方式写入，避免保存中断破坏原文件。

## 日常推荐流程

```text
npm run editor
    ↓
新增或编辑文章
    ↓
保存草稿
    ↓
确认并发布
    ↓
打开 /reading/ 本地预览
    ↓
检查 Git diff
    ↓
提交并推送到 GitHub
```

## 常见问题

### 浏览器没有自动打开

手动访问 <http://127.0.0.1:4173/>。

### 端口 4173 已被占用

可以指定其他端口：

```bash
npm run editor -- --port 4175
```

然后访问 <http://127.0.0.1:4175/>。

### 保存后线上网站没有变化

检查以下事项：

1. 文章是否已经点击“发布”；
2. `data/reading-library.json` 是否已经提交；
3. Git 改动是否已经推送到 GitHub；
4. GitHub Pages 部署是否已经完成。

### 为什么不能通过 GitHub Pages 打开管理工具

这是预期行为。管理工具位于 `_tools/`，并在仓库根目录的 `_config.yml` 中明确排除，不会进入 GitHub Pages 的公开网站。它只能通过本地 Node 服务使用。
