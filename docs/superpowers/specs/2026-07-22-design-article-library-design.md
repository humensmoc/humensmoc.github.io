# 设计文章库功能设计

## 1. 背景与目标

个人网站需要增加一个设计文章策展区，用来整理日常发现的高价值文章。它不是通用内容平台，也不负责转载原文；核心价值是通过主分类、Tag 和个人推荐理由，解释一篇文章为什么值得保留和阅读。

系统包含两个边界清晰的部分：

- 公开阅读端：部署在 GitHub Pages，供访客浏览、搜索和筛选已发布文章。
- 本地采编端：通过本机 Node 服务运行，供站点维护者录入、编辑、分类和发布文章。

第一版继续采用纯静态网站部署，不引入账号、数据库、云端后台或第三方 CMS。

## 2. 已确认的产品原则

- 文章标题由维护者手动填写，不自动抓取网页元数据。
- `summary` 与 `comment` 合并为一个“推荐理由”字段。
- 每篇文章必须属于一个主分类，并可关联零个或多个 Tag。
- 新文章默认保存为草稿，经管理工具中的发布操作后才会出现在公开页面。
- 草稿不是保密数据，可以与已发布文章保存在同一个公开 JSON 中。公开页面不展示草稿，但直接查看 JSON 或 Git 历史仍可能看到它们。
- 管理工具不执行 Git 提交或推送。真正上线仍由维护者在 VS Code 或 Cursor 中完成 Git 操作。

## 3. 总体架构

```text
本地管理页
    ↓ localhost HTTP API
Node 管理服务
    ↓ 校验并原子写入
data/reading-library.json
    ↓ 读取并筛选 published
公开静态阅读页
    ↓ Git 提交和推送
GitHub Pages
```

建议的文件结构：

```text
data/
  reading-library.json

reading/
  index.html
  reading.css
  reading.js

_tools/
  article-manager/
    server.js
    index.html
    manager.js
    manager.css

package.json
```

各组件职责如下：

- `data/reading-library.json` 是分类、Tag、草稿和已发布文章的唯一数据源。
- `reading/` 是公开静态页面，只读取数据并展示 `published` 文章。
- `_tools/article-manager/` 是本地管理工具。下划线目录依赖 GitHub Pages 默认的 Jekyll 排除规则，不进入网站输出。
- Node 服务只监听 `127.0.0.1`，提供管理页面及有限的数据读写 API。
- `package.json` 提供统一入口 `npm run editor`，启动服务后自动打开浏览器。

如果将来改用 `.nojekyll` 或自定义发布流程，必须同步保证 `_tools/` 不进入公开产物。

## 4. 数据结构

初始数据格式如下：

```json
{
  "meta": {
    "schemaVersion": 1,
    "updatedAt": "2026-07-22T10:30:00+08:00"
  },
  "categories": [
    {
      "id": "game-design",
      "name": "游戏设计"
    }
  ],
  "tags": [
    {
      "id": "level-design",
      "name": "关卡设计"
    }
  ],
  "articles": [
    {
      "id": "73eb23da-4770-4f72-aea2-9a3268ce3ee4",
      "url": "https://example.com/article",
      "title": "文章标题",
      "recommendation": "为什么保留并推荐这篇文章……",
      "categoryId": "game-design",
      "tagIds": ["level-design"],
      "status": "draft",
      "createdAt": "2026-07-22T10:00:00+08:00",
      "updatedAt": "2026-07-22T10:30:00+08:00",
      "publishedAt": null
    }
  ]
}
```

数据约束：

- 文章 ID 由管理工具通过 `crypto.randomUUID()` 生成，创建后保持稳定。
- 文章的链接、标题、推荐理由和主分类必填。
- 文章链接只允许 `http` 或 `https`，并且不能重复。
- `status` 只能是 `draft` 或 `published`。
- Tag 可以为空，也可以多选；重复 Tag ID 不允许出现。
- 分类 ID 和 Tag ID 必须指向现有记录。
- 分类和 Tag 名称去除首尾空格后不能为空，并进行不区分大小写的重复检查。
- `createdAt` 在创建时写入；每次保存更新 `updatedAt`。
- 首次发布时写入 `publishedAt`。撤回不会清除首次发布时间，再次发布沿用原时间。
- 公开页面默认按 `publishedAt` 从新到旧排序，JSON 数组顺序不承担排序语义。

## 5. 本地管理工具

### 5.1 启动与状态

运行 `npm run editor` 后，Node 服务监听 `127.0.0.1` 并自动打开管理页。页面顶部显示：

- 数据最后保存时间；
- 草稿数量和已发布数量；
- 当前是否存在未保存修改；
- 最近一次保存操作的成功或失败反馈。

### 5.2 文章管理

文章列表支持按标题搜索，并按状态、主分类和 Tag 筛选。点击列表项进入编辑状态。

文章表单包含：

- 原文链接；
- 标题；
- 推荐理由；
- 单选主分类；
- 可搜索的多选 Tag。

可执行操作：

- 新建并保存草稿；
- 编辑现有文章；
- 发布草稿；
- 将已发布文章撤回为草稿；
- 删除文章。

创建文章时若链接已经存在，工具阻止重复创建并定位到原记录。发布前重新执行完整字段校验。删除文章需要二次确认。编辑已发布文章并保存后会立即改变公开数据文件，但线上网站仍要等 Git 推送和部署完成后才会更新。

当表单有未保存修改时，切换文章、刷新或关闭页面均需提示维护者。

### 5.3 主分类管理

主分类管理支持：

- 新建和重命名分类；
- 查看每个分类关联的文章数量；
- 删除未使用分类；
- 迁移并删除正在使用的分类。

删除未使用分类需要普通二次确认。删除正在使用的分类时必须先选择替代分类，服务端在同一次保存中迁移所有文章并删除旧分类。系统必须始终至少保留一个主分类。

### 5.4 Tag 管理

Tag 管理支持：

- 新建和重命名 Tag；
- 查看每个 Tag 关联的文章数量；
- 将一个 Tag 合并到另一个 Tag；
- 删除 Tag。

未使用 Tag 通过普通二次确认删除。删除正在使用的 Tag 时，界面必须显示受影响文章数量，并要求强化二次确认；确认后在同一次保存中删除 Tag 及所有文章中的对应引用。

合并 Tag 时，源 Tag 的所有文章引用迁移到目标 Tag。若文章已经包含目标 Tag，则结果需要去重，随后删除源 Tag。

## 6. 公开阅读页

页面工作名称为“设计文章库”，后续可以修改展示标题而不影响数据。

公开页包含：

- 页面标题和简短介绍；
- 全文搜索框；
- 主分类导航；
- 多 Tag 筛选；
- 当前筛选条件和结果数量；
- 文章卡片列表；
- 空结果和加载失败状态。

搜索范围包括文章标题、推荐理由、主分类名称和 Tag 名称。主分类为单选，Tag 为多选，并采用“同时满足全部已选 Tag”的交集规则。所有条件可以组合使用。

文章卡片展示：

- 文章标题；
- 推荐理由；
- 主分类；
- Tag；
- 发布时间；
- 从 URL 动态提取的来源域名；
- 在新标签页打开的“阅读原文”操作。

页面采用浏览器端筛选，不做分页。手机端将筛选区变为紧凑或折叠布局。没有匹配结果时提供“清除筛选”操作；数据加载失败时显示明确错误，不呈现无解释的空白页面。

第一版不包含文章详情页、访客账号、收藏、评论、浏览量、自动元数据抓取或多种排序模式。

## 7. 本地 API 边界

API 只服务管理页面，可采用以下最小接口：

- `GET /api/library`：读取完整数据。
- `PUT /api/library`：校验并保存完整数据。

第一版以完整文档读写代替大量细粒度 CRUD 接口，降低服务器复杂度。服务端始终重新校验完整数据，不能信任浏览器提交结果。

写接口必须：

- 仅接受 JSON；
- 检查请求来源为本地管理页同源；
- 限制合理的请求体大小；
- 拒绝未知文件路径和任意文件访问；
- 校验通过后先写临时文件，再原子替换正式文件；
- 写入失败时保留原数据并返回可理解的错误。

## 8. 安全与内容渲染

- Node 服务只监听 `127.0.0.1`，不暴露到局域网。
- 标题、推荐理由、分类和 Tag 均按纯文本渲染，不能通过 `innerHTML` 注入页面。
- 外部文章链接只接受 `http` 和 `https`。
- 新标签页链接使用 `noopener noreferrer`。
- Node 启动时校验现有 JSON。若数据无效，服务可以提供只读错误页，但不得用空数据覆盖原文件。
- 管理页面请求失败时保留当前表单内容，允许维护者重试。

## 9. 测试与验收

优先使用 Node 内置测试能力，避免为了第一版引入大型依赖。

自动测试覆盖：

- 数据结构和引用完整性校验；
- 新建、编辑、发布、撤回和删除文章；
- 重复链接拦截；
- 分类迁移和删除；
- Tag 重命名、合并和删除；
- 写入失败时原文件保持完整；
- 公开端只选择 `published` 文章；
- 标题和推荐理由搜索；
- 主分类筛选；
- 多 Tag 交集筛选；
- 特殊字符按纯文本安全渲染。

人工验收覆盖：

1. 运行 `npm run editor` 后，浏览器能够打开管理页。
2. 在 Chrome 中完成一篇文章从新建草稿到发布的完整流程。
3. 刷新管理页后，保存的数据仍然存在。
4. 公开页面不会展示草稿。
5. 公开页面在桌面和手机宽度下均可操作。
6. `_tools/` 中的管理页面无法通过 GitHub Pages 地址访问。
7. Git 推送并完成部署后，新发布文章出现在公开页面。

## 10. 发布流程

日常使用流程固定为：

1. 在仓库中运行 `npm run editor`。
2. 在本地管理页新增、编辑或发布文章。
3. 在公开页的本地预览中检查展示结果。
4. 在 VS Code 或 Cursor 中检查数据差异。
5. 手动提交并推送 Git 变更。
6. 等待 GitHub Pages 部署完成并检查线上页面。

该流程将内容发布决策放在管理工具中，将网站部署决策保留在现有 Git 工作流中，避免一次点击同时改变内容状态和远端仓库。
